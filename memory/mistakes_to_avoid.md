## 📝 发布流程最佳实践

### ✅ 必须这样做
- **使用微信官方 API** — 不要依赖第三方自定义接口，直接使用微信官方接口
- **使用永久素材接口** — 上传封面图使用 `add_material` 接口，而非临时素材接口
- **使用标准库** — 优先使用 Python 标准库 `urllib`，避免安装依赖问题
- **检查环境变量** — 发布前确认 `WECHAT_APPID` 和 `WECHAT_APPSECRET` 已配置
- **文件编码检查** — 发布前确认HTML文件为UTF-8编码，无BOM或正确包含BOM 🆕
- **AI味检查** — 文章应减少123分段，使用更自然的小标题和过渡 🆕
- **复用已上传封面 media_id** — 同一封面图无需重复上传，直接读取 `cover_media_id.txt` 🆕

### 📁 文件管理规范 🆕新增
- **所有文章定稿必须保存在 `output/` 目录**，禁止保存在根目录
- **命名规范**：`output/article_{topic}_v{version}.html`
  - 例如：`output/article_taipingnian_zhaokuangyin_v5.html`
- **中间版本保留**：v1, v2, v3... 都保留在 output/，作为迭代记录
- **发布脚本统一**：使用根目录的 `publish_draft.py`，从 `output/` 读取文章
- **封面 media_id 复用**：统一读取 `cover_media_id.txt`（根目录）
- **临时脚本清理**：用完即删，避免根目录堆积 `temp_*.py`、`publish_v*.py`

### ❌ 不要这样做
- 依赖可能失效的第三方发布接口
- 使用临时素材接口上传封面（会导致 media_id 无效）
- 发布前不测试 API 连通性
- **不检查文件编码直接发布**（会导致中文乱码） 🆕
- **使用机械的123分段结构**（会被读者识别为AI生成） 🆕
- **将文章HTML文件直接保存在根目录**（会导致文件管理混乱） 🆕

## 📚 引用与原创规范 🆕新增章节

### 绝对禁止
- **大段复制粘贴**他人文章内容（超过100字）
- **整段搬运**其他媒体的报道原文
- **直接剽窃**他人的观点表述而不改写

### 必须遵守
| 情况 | 正确做法 | 错误做法 |
|-----|---------|---------|
| 引用公开言论 | "黄仁勋在2026年GTC大会上表示：'...'" | 直接复制报道原文，不加引号和出处 |
| 使用数据 | "据国家统计局2026年3月公布，..." | 直接搬运数据，不注明来源 |
| 转述观点 | 用自己的话重新组织，并注明"据XX分析" | 一字不差复制他人分析 |
| 新闻事实 | 多源交叉验证，简洁陈述 | 直接复制某一篇文章的表述 |

### 审核检查清单
- [ ] 所有引语是否都标注了说话人和场合？
- [ ] 所有数据是否都注明了来源？
- [ ] 是否存在与其他文章高度相似的段落？
- [ ] 观点部分是否加入了自己的分析和判断？

## 🎭 AI味降低指南 🆕新增章节

### 问题表现
- 使用"一、二、三"或"1、2、3"的机械分段
- 每个段落都是"总-分-总"结构
- 过渡生硬，缺乏情感连接
- 语言过于书面化，缺乏口语感

### 改进方法
1. **去掉123分段**：改用有画面感的小标题
   - ❌ "一、历史背景" "二、核心分析"
   - ✅ "郭荣面对的，是一个死局" "陈桥兵变：一场演砸了的'意外'"

2. **增加互动性表达**：
   - 使用反问句："这是个什么世道？"
   - 使用设问句："黄袍哪来的？"
   - 使用口语化表达："但我想泼盆冷水"

3. **优化段落过渡**：
   - ❌ "下面我们来分析..."
   - ✅ "回到那个虚构的场景"

4. **敢于下判断**：
   - ❌ "有人认为...也有人认为..."
   - ✅ "我的看法是：他就是欺负了孤儿寡母，但他没得选"

5. **增加金句密度**：
   - 每500字至少1个让人记住的句子
   - 金句要有冲击力，不能是套话

## 🔤 文件编码规范 🆕新增章节

### 问题背景
微信公众号API要求文章内容使用**UTF-8编码**。如果文件编码错误，发布后中文会显示为乱码（如"Token濮濓絽婀崥鍐╁竴娴ｇ姷娈戝銉ㄧカ閺夆槄绱?"）。

### 编码要求
- **必须使用UTF-8编码**保存所有HTML文件
- **推荐UTF-8 without BOM**（无字节顺序标记）
- **不要**使用GBK、GB2312、Latin1等编码

### 正确的Python代码示例
```python
# 正确：使用UTF-8编码读取
with open('article.html', 'r', encoding='utf-8') as f:
    content = f.read()

# 正确：使用UTF-8编码写入
with open('article.html', 'w', encoding='utf-8') as f:
    f.write(content)

# 如果文件有BOM，使用utf-8-sig
with open('article.html', 'r', encoding='utf-8-sig') as f:
    content = f.read()
```

### 发布前编码检查清单
- [ ] HTML文件已保存为UTF-8编码
- [ ] 文件中的中文字符能正常显示（无乱码）
- [ ] 使用Python读取文件后，中文内容正确无误
- [ ] JSON序列化时使用 `ensure_ascii=False`

## 📖 历史教训记录

### 2026-04-02 发布流程踩坑：换用不同脚本导致封面上传失败 🆕新增
- **问题**：昨天成功发布的文章使用了 `publish_final.py`（复用已有 cover_media_id + urllib 直调官方API），今天改用了 `skills/wechat-publish/scripts/publish_wechat.py` 的 `workflow` 模式，该模式会尝试重新上传封面图，结果踩到两个坑：
  1. 微信临时素材接口 `media/upload` 对 `thumb` 类型有 **64KB 大小限制**，`default.jpg`（222KB）上传失败，报错 `40006 invalid media size`
  2. 当前工作环境的 Python 没有预装 `requests` 和 `pip`，安装过程曲折耗时
- **后果**：发布流程中断近20分钟，用户体验下降
- **改进**：
  1. **优先复用已上传的封面 media_id**：直接读取 `cover_media_id.txt`，避免重复上传
  2. **使用稳定的发布脚本**：沿用已验证成功的 `publish_final.py` 模式
  3. **优先使用 Python 标准库 `urllib`**：避免在未知环境中安装第三方依赖
  4. 如需上传新封面，使用微信 `material/add_material` 永久素材接口，而非 `media/upload` 临时素材接口
