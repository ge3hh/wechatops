## 📝 引用与原创规范 ⭐新增章节

### 绝对禁止
- **大段复制粘贴**他人文章内容（超过100字）
- **整段搬运**其他媒体的报道原文
- **直接抄袭**他人的观点表述而不改写

### 必须遵守
| 情况 | 正确做法 | 错误做法 |
|-----|---------|---------|
| 引用公开言论 | "黄仁勋在2026年GTC大会上表示：'...'" | 直接复制报道原文，不加引号和出处 |
| 使用数据 | "据国家数据局2026年3月公布，..." | 直接搬运数据，不注明来源 |
| 转述观点 | 用自己的话重新组织，并注明"据XX分析" | 一字不差复制他人分析 |
| 新闻事实 | 多源交叉验证，简洁陈述 | 直接复制某一篇报道的表述 |

### 审核检查清单
- [ ] 所有引语是否都标注了说话人和场合？
- [ ] 所有数据是否都注明了来源？
- [ ] 是否存在与其他文章高度相似的段落？
- [ ] 观点部分是否加入了自己的分析和判断？

## 🔧 文件编码规范 ⭐新增章节

### 问题背景
微信公众号API要求文章内容使用**UTF-8编码**。如果文件编码错误，发布后中文会显示为乱码（如"Token姝ｅ湪鍚冩帀浣犵殑宸ヨ祫鏉★細"）。

### 编码要求
- **必须使用UTF-8编码**保存HTML文件
- **推荐使用UTF-8 without BOM**（无字节顺序标记）
- **禁止**使用GBK、GB2312、Latin1等编码

### 正确的Python代码示例
```python
# 正确：使用UTF-8编码读取
with open('article.html', 'r', encoding='utf-8') as f:
    content = f.read()

# 正确：使用UTF-8编码写入
with open('article.html', 'w', encoding='utf-8') as f:
    f.write(content)

# 如果文件有BOM，使用utf-8-sig
with open('article.html', 'r', encoding='utf-8-sig') as f:
    content = f.read()
```

### 编码检查方法
```python
# 检查文件编码
import chardet

with open('article.html', 'rb') as f:
    raw_data = f.read()
    result = chardet.detect(raw_data)
    print(f"文件编码: {result['encoding']}")
```

### 发布前编码检查清单
- [ ] HTML文件已保存为UTF-8编码
- [ ] 文件中的中文字符能正常显示（无乱码）
- [ ] 使用Python读取文件后，中文内容正确无误
- [ ] JSON序列化时使用 `ensure_ascii=False`

## 📚 历史踩坑记录

### 2026-03-29 文章颜色使用过多导致版面花哨 ⭐新增
- **问题**：Token经济文章中使用了5种不同颜色（浅黄、浅绿、浅红、浅蓝、浅紫）来标记不同类型的信息，共24处高亮。用户反馈版面显得"花"，影响阅读体验
- **后果**：
  - 颜色过多分散读者注意力
  - 读者不理解颜色语义系统，造成困惑
  - 版面显得杂乱，不够简洁专业
- **改进**：
  1. 简化配色方案：整篇文章高亮颜色不超过2-3种
  2. 减少高亮密度：关键数据/观点使用浅黄色（5-8处），警示信息使用浅红色（2-3处），其他内容使用**加粗**即可
  3. 不期望读者理解颜色语义，颜色仅作为辅助
  4. 在 style_guide.md 中更新配色方案章节，强调"少即是多"原则
  5. 在 reviewer 审核清单中增加"颜色使用检查"：是否颜色过多？是否影响阅读？

### 2026-03-29 文章编码错误导致中文乱码 ⭐新增

- **改进**：

  2. 使用Python的 `encoding='utf-8'` 参数读取和写入文件
  3. 发布前本地验证中文显示正常
  4. 在 mistakes_to_avoid.md 中增加"文件编码规范"章节
